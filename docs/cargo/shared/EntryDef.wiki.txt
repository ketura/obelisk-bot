A generic, hand-curated reference table for **named terms with no extra structure**. Every row carries:

* a '''type''' + '''subtype''' primary key
* a player-facing '''display_name''' + '''description''' (English defaults)
* an optional '''narrative_description''' for flavor/lore text (D-036)
* the canonical L10n '''name_sid''' + '''desc_sid''' + optional '''narrative_description_sid''' the row was sourced from
* an optional '''icon''' filename
* a '''source_path''' pointer to the L10n file (or source JSON for per-patch extracts) the row came from (D-036)
* 15 × per-language '''<code><lang>_name</code>''' + '''<code><lang>_desc</code>''' columns

This table is the catch-all for any reference data that fits the "name + description + i18n" shape and nothing else. As soon as an entity needs ''additional'' columns (rank, pattern_token, cost, prerequisites, etc.) it belongs in its own dedicated table.

Entry rows come from two sources:

# '''Hand-curated seeds''' — small enums the wiki ships pre-populated (attack archetypes, movement types, creature types). These live in the <code>ENTRY_SEEDS</code> dict in <code>src/obelisk/emit/unit.py</code>.
# '''Per-patch extracted data''' — name+i18n-only data harvested from the source JSON each patch (currently: faction city names from <code>DB/fractions/*.json</code>). The extractor module (<code>extract/faction.py</code>, etc.) hands the SID directly to <code>emit_entry_page</code>; no seed-dict detour.

In both cases the bot resolves each row's translations from the L10n corpus on every extract, so per-patch text changes flow through.

== Field notes ==
* '''Primary key is <code>(type, subtype, variant)</code>''' as a tuple. <code>variant</code> is sparse — it's set for sub-row translations (<code>type='law_level'</code>, <code>type='skill_level'</code>) where the same <code>(type, subtype)</code> pair needs to disambiguate per-level rows, and left unset for everything else. Joins from per-unit columns include the type discriminator: e.g. <code>UnitDef.attack_type</code> joins to <code>EntryDef.subtype WHERE EntryDef.type='attack_archetype'</code>.
* '''<code>variant</code>''' carries sub-row indices for rows that fold per-(entity, level) data into this table — currently <code>type='law_level'</code> and <code>type='skill_level'</code> (folded back from the now-removed <code>LawLevelTranslationDef</code> and <code>SkillLevelTranslationDef</code>). String-typed so it can hold either integer levels (<code>"1"</code>, <code>"2"</code>, <code>"3"</code>) or freeform variant codes if future sub-rows need them.
* '''<code>display_name</code> / <code>description</code>''' carry the English defaults inline. Translations live in the per-language columns on the same row. (No separate translation table — D-024 collapsed the <code><Entity></code> / <code><Entity>Translation</code> pair into one table per row.)
* '''<code>icon</code>''' is optional. Sparse-emit: if a seed has no icon configured, the field is omitted from the row.
* '''<code>name_sid</code> / <code>desc_sid</code>''' are surfaced for traceability — they point at the L10n entries the row was sourced from. Useful for "find which game string drives this label" queries.

== Page layout ==
Each Entry <code>type</code> gets its own top-level wiki namespace and on-disk directory — the <code>EntryDef</code> Cargo table name only surfaces in the table schema and the <code><nowiki>{{EntryDef | …}}</nowiki></code> template invocations ''inside'' each page. The <code>EntryDef</code> abstraction is internal; user-facing wiki paths use the per-domain names that already exist as game concepts.

* '''On wiki:''' <code>Data:<PascalType>/<subtype></code> (e.g. <code>Data:Movement/fly</code>).
* '''On disk:''' <code>data/<type>/<subtype>.wiki.txt</code> (e.g. <code>data/movement/fly.wiki.txt</code>).

Examples:

{| class="wikitable"
! Type !! On disk !! On wiki
|-
| <code>attack_archetype</code> || <code>data/attack_archetype/melee.wiki.txt</code> || <code>Data:AttackArchetype/melee</code>
|-
| <code>movement</code> || <code>data/movement/fly.wiki.txt</code> || <code>Data:Movement/fly</code>
|-
| <code>creature_type</code> || <code>data/creature_type/demon.wiki.txt</code> || <code>Data:CreatureType/demon</code> (Hive Spawn)
|-
| <code>FactionCityName</code> || (inline on <code>data/factions/<id>.wiki.txt</code>) || (inline on <code>Data:Faction/<id></code>)
|}

The directory→wiki-namespace mapping lives in <code>_DIR_TO_WIKI_TABLE</code> (<code>src/obelisk/diff/wiki_diff.py</code>) — extend it when adding a new Entry type.

<code>display_name</code> and <code>description</code> come from the L10n corpus via
<code>name_sid</code> / <code>desc_sid</code> on every extract; the table below summarizes.

=== <code>type=FactionCityName</code> (120 rows; per-patch extracted) ===
Six factions × 20 names each. Subtype is <code><faction>_<index></code> (e.g.
<code>dungeon_1</code> … <code>dungeon_20</code>). Source: <code>cityNames</code> array on each
faction record in <code>DB/fractions/*.json</code>. Display names are genuinely translated (8-16 distinct strings per name across the 16 languages — CJK languages get full character-set translations; Latin scripts mix preservation, idiomatic translation, and phonetic transliteration). '''No <code>desc_sid</code> exists for city names''' — description columns are sparse-emitted. See D-025.

Unlike the rows above, these are not hand-curated seeds — they're extracted per-patch from the source JSON. They also do ''not'' live on individual wiki pages: the 20 city Entry rows for a given faction are appended to that faction's <code>Data:Faction/<id></code> page (after the
<code><nowiki>{{FactionDef}}</nowiki></code> and <code><nowiki>{{FactionTranslation}}</nowiki></code> blocks, in numeric source
order). Cargo stores the rows in the unified <code>EntryDef</code> table just the same; the page-name namespace <code>Data:FactionCityName/…</code> simply isn't used. The faction emitter calls <code>render_entry_block(...)</code> to inline each row.

== What does NOT belong here ==
Anything with structure beyond name/description/i18n keeps its own table. For example, <code>AttackPassiveDef</code> carries <code>pattern_token</code> and <code>rank</code> columns and stays separate. The rule: if you'd add a column for a single use case, it goes in its own dedicated table. Entry is for the "identical shape, different domain" cases only.
