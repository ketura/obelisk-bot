A generic, hand-curated reference table for '''named terms with no extra structure''' — the catch-all for any reference data that fits the "name + description + i18n" shape and nothing else. As soon as an entity needs ''additional'' columns (rank, pattern_token, cost, prerequisites, …) it belongs in its own dedicated table.

Entry is a '''thin structural stub'''. It carries only the identity and traceability of each term; all display text — names, descriptions, and their translations, English included — lives in the shared [[Template:TranslationDef|Translation]] table, whose <code>(type, subtype, variant)</code> key tiers match this row's <code>(type, subtype, variant)</code>.

Each row carries:
* a '''type''' + '''subtype''' + '''variant''' primary key
* an optional '''icon''' filename
* the canonical L10n '''name_sid''' + '''desc_sid''' + optional '''narrative_description_sid''' the row was sourced from — traceability only, not part of the lookup path
* a '''source_path''' pointer to the L10n file (or source JSON for per-patch extracts) the row came from

Entry rows come from two sources:

# '''Hand-curated seeds''' — small enums the wiki ships pre-populated (attack archetypes, movement types, creature types). These live in the <code>ENTRY_SEEDS</code> dict in <code>src/obelisk/emit/unit.py</code>.
# '''Per-patch extracted data''' — name+i18n-only data harvested from the source JSON each patch (currently: faction city names from <code>DB/fractions/*.json</code>). The extractor module (<code>extract/faction.py</code>, etc.) hands the SID directly to <code>emit_entry_page</code>; no seed-dict detour.

In both cases the bot resolves each row's translations from the L10n corpus on every extract and emits them as <code>Translation</code> rows, so per-patch text changes flow through.

== Field notes ==
* '''Primary key is <code>(type, subtype, variant)</code>''' as a tuple. <code>variant</code> is the optional third key tier — sparse, and not used by any Entry type that currently ships (it's kept on the stub for uniformity with the shared key pyramid). Joins from per-unit columns include the type discriminator: e.g. <code>UnitDef.attack_type</code> joins to <code>EntryDef.subtype WHERE EntryDef.type='attack_archetype'</code>.
* '''<code>variant</code>''' is String-typed so it can hold either integer indices or freeform variant codes should a future Entry type need a third key tier. Per-(entity, level) data — law levels, skill levels, spell ranks — does '''not''' fold into Entry; those are per-level rows of real parent entities and live in their own structural tables ([[Template:LawLevelDef|LawLevel]], [[Template:SkillLevelDef|SkillLevel]], [[Template:SpellRankDef|SpellRank]]) with translations keyed by <code>target_id</code> + <code>variant</code> in the [[Template:TranslationDef|Translation]] table.
* '''All display text is in [[Template:TranslationDef|Translation]]''', not here. To get the English label for an Entry term: <code>Translation.name WHERE type='<type>' AND subtype='<subtype>' AND variant='<variant>' AND language='en'</code>. Other languages swap the <code>language</code> filter.
* '''<code>icon</code>''' is optional. Sparse-emit: if a seed has no icon configured, the field is omitted from the row.
* '''<code>name_sid</code> / <code>desc_sid</code> / <code>narrative_description_sid</code>''' are surfaced for traceability — they point at the L10n entries the row was sourced from. Useful for "find which game string drives this label" queries, but they are not the join path to translations.

== Page layout ==
Each Entry <code>type</code> gets its own top-level wiki namespace and on-disk directory — the <code>EntryDef</code> Cargo table name only surfaces in the table schema and the <code><nowiki>{{EntryDef | … }}</nowiki></code> template invocations ''inside'' each page. The <code>EntryDef</code> abstraction is internal; user-facing wiki paths use the per-domain names that already exist as game concepts.

* '''On wiki:''' <code>Data:<PascalType>/<subtype></code> (e.g. <code>Data:Movement/fly</code>).
* '''On disk:''' <code>data/<type>/<subtype>.wiki.txt</code> (e.g. <code>data/movement/fly.wiki.txt</code>).

Each Entry page emits the <code><nowiki>{{EntryDef}}</nowiki></code> stub row followed by one <code><nowiki>{{TranslationDef}}</nowiki></code> row per language.

Examples:

{| class="wikitable"
! Type !! On disk !! On wiki
|-
| <code>attack_archetype</code> || <code>data/attack_archetype/melee.wiki.txt</code> || <code>Data:AttackArchetype/melee</code>
|-
| <code>movement</code> || <code>data/movement/fly.wiki.txt</code> || <code>Data:Movement/fly</code>
|-
| <code>creature_type</code> || <code>data/creature_type/demon.wiki.txt</code> || <code>Data:CreatureType/demon</code> (Hive Spawn)
|-
| <code>FactionCityName</code> || (inline on <code>data/factions/<id>.wiki.txt</code>) || (inline on <code>Data:Faction/<id></code>)
|}

The directory→wiki-namespace mapping lives in <code>_DIR_TO_WIKI_TABLE</code> (<code>src/obelisk/diff/wiki_diff.py</code>) — extend it when adding a new Entry type.

=== <code>type=FactionCityName</code> (120 rows; per-patch extracted) ===
Six factions × 20 names each. Subtype is <code><faction>_<index></code> (e.g. <code>dungeon_1</code> … <code>dungeon_20</code>). Source: <code>cityNames</code> array on each faction record in <code>DB/fractions/*.json</code>. Display names are genuinely translated (8-16 distinct strings per name across the 16 languages — CJK languages get full character-set translations; Latin scripts mix preservation, idiomatic translation, and phonetic transliteration). '''No <code>desc_sid</code> exists for city names''' — description translation rows are simply not emitted.

Unlike the rows above, these are not hand-curated seeds — they're extracted per-patch from the source JSON. They also do ''not'' live on individual wiki pages: the 20 city Entry stubs for a given faction (plus their Translation rows) are appended to that faction's <code>Data:Faction/<id></code> page, in numeric source order. The faction emitter calls <code>render_entry_block(...)</code> to inline each row.

== What does NOT belong here ==
Anything with structure beyond name/description/i18n keeps its own table. For example, <code>AttackPassiveDef</code> carries <code>pattern_token</code> and <code>rank</code> columns and stays separate. The rule: if you'd add a column for a single use case, it goes in its own dedicated table. Entry is for the "identical shape, different domain" cases only.
