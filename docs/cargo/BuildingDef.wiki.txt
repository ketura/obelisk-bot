One row per (faction, building, level) — '''206 rows''' in the 2026-05-05 corpus across 6 factions and 18 building categories. The source's <code>parametersPerLevel</code> array is ''flattened'' on extract: a 3-level building like Build_Main produces 3 separate Building rows (<code><faction>_Build_Main_L1</code>, <code>_L2</code>, <code>_L3</code>).

A building's per-level ''passive bonuses'' (<code>bonusesPerLevel</code>), ''optional effects'' (<code>optionalEffectsPerLevel</code>), magic-guild ''spell roll chances'' (<code>rollChances</code>), and most other category-specific extras are '''deliberately not extracted''' in this pass — they'd need their own decision/table and the value-to-effort ratio is currently low. Creature-dwelling unit data (<code>unitsHire</code>) IS pulled in, simplified to a single base-unit SID + weekly increment.

== Page layout ==
<code>Data:Building/<id></code> carries:

# <code><nowiki>{{BuildingDef | id=<faction>_<sid>_L<level> | faction=… | sid=… | level=… | name=… | desc=… | …}}</nowiki></code>
# <code><nowiki>{{TranslationDef | type=building | target_id=<id> | <lang>_name=… | <lang>_desc=… | …}}</nowiki></code>

No side rows — all per-level data flattens onto the Building row itself; multi-resource costs spread across fixed <code><resource>_cost</code> columns; prerequisite list goes into a <code>List</code> column queryable via Cargo's <code>HOLDS</code> operator.

== Field notes ==
* '''Primary key is <code>id</code>''', synthesized as <code><faction>_<sid>_L<level></code> (e.g. <code>human_Build_Main_L1</code>). Building SIDs are unique within a faction but reused across factions (every faction has <code>Build_Main</code>), so the faction prefix is needed for global uniqueness.
* '''<code>category</code>''' is the source group key the building came from (one of: <code>mains</code>, <code>walls</code>, <code>magicGuilds</code>, <code>taverns</code>, <code>markets</code>, <code>graals</code>, <code>banks</code>, <code>hires</code>, <code>artifactMarkets</code>, <code>heroBonusBanks</code>, <code>intelligences</code>, <code>manaFountains</code>, <code>myceliumRoots</code>, <code>portalSummonings</code>, <code>rebirthShrines</code>, <code>trainingRanges</code>, <code>unitsConverters</code>, <code>artifactChangers</code>). The single-instance specials (intelligences, manaFountains, etc.) are faction-unique buildings — <code>category</code> lets the wiki style/group them.
* '''<code>level</code>''' is 1-based; query <code>WHERE sid=<x> AND level=<n></code> to pull a specific tier of a building, or <code>WHERE sid=<x></code> for all tiers in numeric order.
* '''<code>name</code> / <code>desc</code>''' carry English defaults inline; per-language values live on the matching <code><nowiki>{{TranslationDef | type=building}}</nowiki></code> row.
* '''<code>prereqs</code>''' is a Cargo <code>List</code> of <code><sid>_L<level></code> strings (e.g. <code>Build_Main_L1,Build_Tier_2_L1</code>). Empty list when no prereq. Query "what unlocks at Build_Main level 2" via <code>WHERE prereqs HOLDS 'Build_Main_L2'</code>.
* '''<code><resource>_cost</code>''' columns are sparse — each is NULL unless the source's <code>costs</code> array includes that resource for this level. Eight columns total (<code>gold/wood/ore/crystals/gemstones/mercury/dust/graal</code>). One pre-set column per resource keeps queries scalar-clean ("WHERE gold_cost > 5000").
* '''<code>units_hire_sid</code>''' is the ''base'' unit SID from <code>unitsHire.units[0].sids[0]</code>. The upg / upg_alt variants are reachable via <code>Unit.upgrade_of</code> / equivalent join on the Unit table — no need to denormalize them onto Building.
* '''<code>units_weekly</code>''' is <code>unitsHire.units[0].weeklyIncrement</code>. The dwellings in 2026-05-05 only ever ship a single units-group entry per building, so we collapse it.

== Deliberately not extracted ==
* '''<code>bonusesPerLevel</code>''' — per-level passive bonuses on the building (e.g. mains' <code>+500 gold daily</code>). The data exists in source but isn't currently emitted; would extend the unified <code>BonusDef</code> table with <code>parent_type='building_level'</code> if added later. Out of scope for this pass.
* '''<code>optionalEffectsPerLevel</code>''' — the choose-one effects you pick when upgrading certain buildings (e.g. mains level 2 offers gold/law/astrology). Same story — would need its own side table.
* '''<code>rollChances</code>''' on magic guilds — the per-spell offering weights. ~60 entries per guild, would explode into a <code>BuildingGuildSpell(building_id, spell_sid, chance)</code> side table; deferred.
* '''Wall <code>bonuses</code>, market <code>extraChargePurchase</code> / <code>extraChargeSell</code> / <code>numberPurchases</code> / <code>resArr</code>, training <code>trainingStats</code>, unitsConverter <code>conversionPairs</code>, artifactMarket <code>artifacts</code> / <code>itemsCountPerRarity</code> / <code>levelStep</code> / <code>numberPurchases</code>, graal <code>graalType</code>''' — category-specific data the player rarely queries from a wiki perspective. All deferred.
* '''<code>buildOrders</code>''' — 60 build-order presets used by AI / random map generation, not player-facing buildings. Not extracted (per project decision; noted for hypothetical future work as <code>BuildingPreset</code>).
