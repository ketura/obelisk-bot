One row per hero (~177 in the 2026-05-03 corpus: 108 faction + 46 campaign + 21 tutorial + 2 custom-map). Carries the per-hero identity (name/motto/desc), the link to the parent [[Template:HeroClassDef|HeroClass]], and any class-default fields the hero diverges on. Faction heroes match their class template exactly, so their override fields are sparse-emitted (absent); campaign and tutorial heroes routinely override <code>stats</code>, <code>start_level</code>, and
<code>attacks_times_before</code>.

The bot emits these per-hero blocks at <code>Data:Hero/<id></code>:

# <code><nowiki>{{HeroDef | id=… | name=… | motto=… | desc=… | start_skills=… | start_magics=… | …}}</nowiki></code> — main row. <code>start_skills</code> and <code>start_magics</code> are comma-joined SID lists inline on this row (no side tables for them).
# <code><nowiki>{{TranslationDef | type=hero | target_id=<id> | …}}</nowiki></code> — name/desc translation (per D-026).
# <code><nowiki>{{TranslationDef | type=hero_motto | target_id=<id> | …}}</nowiki></code> — motto translation (separate row, kept off the schema-specialization path; per D-027).
# <code><nowiki>{{HeroStartSquadDef | …}}</nowiki></code> × N — one per starting squad slot (primary + alt collapsed via <code>variant</code> discriminator).

== Override semantics ==
Every Cargo query that wants the effective value of a class-shared field on a hero should use:

<syntaxhighlight lang="sql">
COALESCE(Hero.<col>, HeroClass.<col>) AS effective_<col>
</syntaxhighlight>

Joining on <code>Hero.class_id = HeroClass.id</code>. Hero wins if present; HeroClass fills in otherwise.

For faction heroes (108): all 11 class-shared columns are absent on the Hero row — every query falls through to HeroClass.

For campaign/tutorial heroes: ~3-5 columns are typically present (empirically: <code>stats_*</code>, <code>start_level</code>, <code>attacks_times_before</code>). Other class-shared columns still inherit from HeroClass.

== Field notes ==
* '''<code>id</code>''' is the hero's source JSON id (e.g. <code>human_hero_1</code>, <code>campaign_M10_hero_demon_1</code>).
* '''<code>name_sid</code>/<code>motto_sid</code>/<code>desc_sid</code>''' carry the source SID for traceability and as the join key to the Translation rows. Faction heroes use implicit conventions (<code><id></code> / <code><id>_motto</code> / <code><id>_description</code>); campaign heroes set them explicitly in source JSON.
* '''<code>name</code>/<code>motto</code>/<code>desc</code>''' carry the resolved English text inline. Translations live on the two <code>TranslationDef</code> rows (one for name+desc, one for motto).
* '''<code>class_id</code>''' is <code><class_type>_<faction></code> — matches <code>HeroClass.id</code>.
* '''<code>faction</code>/<code>class_type</code>''' are denormalized from the class so that "all human heroes" / "all might heroes" filters don't need a join.
* '''<code>specialization_id</code>''' points at a row in <code>HeroSpecializationDef</code> (one specialization per hero, 1:1 ratio). Schema for that table comes in a follow-up.

== Related tables ==
* [[Template:HeroClassDef|HeroClass]] — joined on <code>class_id = id</code> (N:1). Carries the class defaults the Hero row may override.
* [[Template:HeroStartSquadDef|HeroStartSquad]] — joined on <code>hero_id = id</code> (1:N). Each row is one slot in primary or alt starting army.
* <code>SkillDef</code> (future) — joined on <code>start_skills</code> LIKE <code>%<sid>%</code> or via Cargo's <code>HOLDS</code> operator on the List column.
* <code>SpellDef</code> (future) — same join pattern via <code>start_magics</code>.
'' [[Template:HeroSpecializationDef|HeroSpecialization]] — joined on <code>specialization_id = id</code> (N:1). The hero's unique passive. ''(Schema lands in a follow-up.)*
* [[Template:TranslationDef|<code>TranslationDef</code>]] — two rows per hero (<code>type=hero</code> and <code>type=hero_motto</code>).
